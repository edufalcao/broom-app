# Broom 1.3.0 Release Plan

> **Date:** 2026-03-17
> **Status:** In Progress
> **Target:** 1.3.0

---

## Summary

`1.3.0` should focus on trust:

- leftover detection should stop surfacing live app data
- uninstall preview should match what will actually be removed
- completion states should clearly report anything Broom skipped or blocked
- the test setup should move from "many unit tests" to "measurable coverage plus end-to-end confidence"

This is a minor release centered on uninstall trust, leftover accuracy, and test infrastructure.

---

## Why This Release

Current code quality is solid in the policy and service layers, and the full suite now passes locally at `176 tests across 27 suites`. The biggest remaining gaps are:

- false positives in leftover detection for app migration cases
- false positives in leftover detection for embedded helper bundle IDs and extension-owned state
- uninstall UI trust gaps between what the user selects and what the plan actually removes
- blocked deletion results not being surfaced to the user
- no UI test target and no coverage reporting in CI

---

## Key Confirmed Issues

### Microsoft Teams false-positive leftover detection

Local validation on **2026-03-17** found:

- Installed app: `/Applications/Microsoft Teams.app`
- Installed bundle ID: `com.microsoft.teams2`
- Leftover candidate still surfaced: `~/Library/Group Containers/UBF8T346G9.com.microsoft.teams`

The candidate is not an orphan:

- its container metadata identifies `com.microsoft.teams2` as the creator
- its files were last modified on **2026-01-09**
- it belongs to the currently installed app family even though the container identifier still uses the older `com.microsoft.teams` name

### Root cause

`OrphanDetector` suppresses candidates using strict name-based bundle ID matching only. That works for:

- `com.microsoft.teams` vs `UBF8T346G9.com.microsoft.teams`

But it does **not** work for:

- installed `com.microsoft.teams2`
- candidate `UBF8T346G9.com.microsoft.teams`

So Broom currently misses migration and compatibility container ownership.

### Required fix

- Read `.com.apple.containermanagerd.metadata.plist` for Group Containers and Application Scripts candidates.
- Suppress candidates when `MCMMetadataCreator` matches an installed or running bundle ID.
- Keep the existing strict name-based matcher as a fast first pass.
- Add a regression test for installed `com.microsoft.teams2` with candidate `UBF8T346G9.com.microsoft.teams`.

### Apple-managed group container false positives

The screenshot review from **2026-03-17** shows generic leftover results for Apple-managed group containers such as:

- `group.com.apple.CoreSpeech`
- `group.com.apple.storekit`
- `group.com.apple.siri.sirisuggestions`
- `group.com.apple.managedappdistributionagent`
- `group.com.apple.mobiletimerd`
- `group.com.apple.poirot.MessagesAnalyticsWorker`

These should **not** appear in "App Leftovers".

### Root cause

Current suppression checks protect names that start with `com.apple.`, but group containers like `group.com.apple.storekit` do not start with that prefix. They currently slip past both the built-in exclusion rules and the protected-data policy.

### Required fix

- Add explicit suppression for Apple-managed group-container namespaces, at minimum:
  - `group.com.apple.*`
- Treat Apple-managed group containers as system-owned data, never as generic leftovers.
- Add regression tests covering the Apple namespaces visible in the screenshot.

### Embedded helper bundle false positives

Local validation on **2026-03-17** found Microsoft AutoUpdate cache entries still being listed as leftovers:

- `~/Library/Caches/com.microsoft.autoupdate.fba`
- `~/Library/HTTPStorages/com.microsoft.autoupdate.fba`

Those are not orphans:

- installed app: `/Library/Application Support/Microsoft/MAU2.0/Microsoft AutoUpdate.app`
- installed top-level bundle ID: `com.microsoft.autoupdate2`
- embedded helper bundle IDs inside the app bundle: `com.microsoft.autoupdate.fba`
- the candidate directories contain live SQLite sidecar files updated on **2026-03-17**

### Root cause

`OrphanDetector` only compares candidate names against the top-level installed bundle IDs in the snapshot. It does not account for helper or embedded bundle identifiers shipped inside installed apps.

### Required fix

- Expand snapshot bundle-ID discovery to include embedded helper bundles inside installed apps.
- Suppress candidates when their reverse-DNS identifier matches any embedded installed helper bundle ID.
- Add regression coverage for installed `com.microsoft.autoupdate2` with embedded helper `com.microsoft.autoupdate.fba`.

### Active descendant false positives

Local validation on **2026-03-17** found `~/Library/Application Support/com.openai.chat` listed as a likely orphan even though it is still active on this machine.

The candidate is not stale:

- it contains `app_pairing_extensions` entries for Cursor and Visual Studio Code
- descendant entries were modified on **2026-03-17**
- only the top-level directory timestamp was older

### Root cause

The stale-age gate currently uses the modification date of the candidate directory itself, not the newest descendant activity inside that directory. That causes live state folders with older parent timestamps to be misclassified as leftovers.

### Required fix

- Compute stale-age using the newest descendant modification date, not just the top-level directory timestamp.
- Apply this to all directory-style leftover candidates, especially caches, HTTP storages, and application-support folders.
- Add regression coverage for a candidate whose parent directory is old but whose descendant files were modified recently.

### Apple account preference false positives

Local validation on **2026-03-17** found `~/Library/Preferences/MobileMeAccounts.plist` listed under lower-confidence leftovers.

That file is not an orphan:

- it stores active Apple account and iCloud service state
- it was modified on **2026-01-15**
- it should never appear in generic app-leftover cleanup

### Root cause

`MobileMeAccounts.plist` did not match the existing system bundle-prefix or cache protections, so it could still flow into orphan detection as a stale preference file.

### Required fix

- Add explicit suppression for Apple-managed account preference files, starting with:
  - `MobileMeAccounts.plist`
- Add regression coverage so these files never appear in generic leftover results.

### Container-only vendor storage showing with too much confidence

Local validation on **2026-03-17** found vendor-scoped group containers such as `6H4HRTU5E3.group.com.avast.osx` appearing as `Likely orphaned` even when the machine still had installed software from the same vendor family.

These entries may be real leftovers, but group-container-only evidence is weaker than:

- receipts
- saved-state matches
- direct installed-owner metadata

### Root cause

Confidence scoring treated any reverse-DNS style name as at least medium confidence, even when the only surviving signal was a group-container identifier.

### Required fix

- Downgrade container-only candidates to lower confidence unless stronger ownership or receipt signals exist.
- Keep higher confidence reserved for candidates with receipts, saved-state matches, or multiple independent signals.

---

## Goals

- Eliminate known live-app and system-data leftover false positives, starting with Teams, Apple-managed group containers, embedded helper bundle IDs, and active descendant state folders.
- Make uninstall selection truthful and editable end to end.
- Surface blocked or skipped deletions clearly in completion UI.
- Add release-grade coverage and UI smoke tests for the main workflows.

## Non-Goals

- New cleaner categories
- Mole-style optimizer features
- CLI work in this release
- Large design refresh unrelated to correctness or testability

## Workstreams

### Workstream 1: Leftover Accuracy Hardening

#### Objective

Make generic leftover detection more conservative for migrated apps, helper bundles, extension-owned state, group containers, and compatibility storage.

#### Status Update

Implemented on the current branch:

- Group Containers and Application Scripts candidates now read `.com.apple.containermanagerd.metadata.plist` and suppress entries owned by installed or running apps.
- Apple-managed `group.com.apple.*` containers are now suppressed.
- Installed-app snapshots now include embedded helper bundle identifiers from app bundle hierarchies.
- Stale-age evaluation now uses the newest descendant modification date for directory candidates.
- `MobileMeAccounts.plist` is now treated as protected Apple account data.
- Container-only candidates now default to lower confidence unless stronger signals exist.
- Regression coverage was added for Teams migration, Apple group containers, embedded helper bundle IDs, recent descendant activity, Apple account preferences, and container-only confidence.

#### Tasks

- Done:
  - Add container metadata reading for:
    - `~/Library/Group Containers/*`
    - `~/Library/Application Scripts/*`
  - Add hard suppression for Apple-managed system group-container namespaces such as:
    - `group.com.apple.*`
  - Expand installed-app snapshot discovery to include embedded helper bundle identifiers inside installed app bundles.
  - Suppress candidates when container metadata shows ownership by an installed or running app.
  - Suppress candidates when their bundle ID matches an embedded helper bundle shipped by an installed app.
  - Keep strict bundle ID matching in place for normal reverse-DNS and team-ID-prefixed matches.
  - Change stale-age evaluation to use the newest descendant modification date for directory candidates.
  - Add explicit suppression for Apple-managed account preferences:
    - `MobileMeAccounts.plist`
  - Downgrade container-only candidates to lower confidence when no stronger signal exists.
  - Add explicit regression fixtures for:
    - `com.microsoft.teams2` installed + `UBF8T346G9.com.microsoft.teams` candidate
    - `group.com.apple.storekit`
    - installed `com.microsoft.autoupdate2` + embedded helper `com.microsoft.autoupdate.fba`
    - old parent directory + recent descendant activity
    - creator metadata ownership
    - Apple account preference files
    - container-only confidence scoring
- Remaining:
  - Add more Apple-managed namespace regressions beyond the currently covered examples.
  - Consider stronger vendor-family suppression for installed apps that ship shared frameworks without direct container metadata.
  - Decide whether low-confidence container-only entries should be hidden by default in the UI.

#### Primary files

- `Broom/Services/AppInventory.swift`

- `Broom/Utilities/BundleIDMatcher.swift`
- `BroomTests/AppInventoryTests.swift`

### Workstream 2: Uninstall Preview Trust

#### Objective

Ensure the detail screen, confirmation sheet, and actual uninstall plan all describe the same file set.

#### Tasks

- Stop showing an uninstall detail view driven by one file source and executing a broader hidden plan built elsewhere.
- Move to a single editable uninstall plan model that:
  - includes planner-discovered artifacts
  - preserves artifact source groups
  - supports per-file selection all the way into execution
- Fix mixed-state group toggles so group actions set an explicit target state instead of toggling every row.
- Keep the uninstall completion screen persistent until the user dismisses it.
- Fix the duplicated "ago" label in the app detail header.

#### Primary files

- `Broom/Services/AppUninstaller.swift`
- `Broom/ViewModels/UninstallerViewModel.swift`
- `Broom/Views/Uninstaller/AppDetailView.swift`
- `Broom/Views/Uninstaller/UninstallConfirmView.swift`
- `Broom/Views/Uninstaller/UninstallerView.swift`

### Workstream 3: Result Reporting and Trust Signals

#### Objective

Make it obvious when Broom did not remove everything it attempted to process.

#### Tasks

- Surface `itemsBlocked` separately from `itemsFailed` in cleaner and uninstall completion states.
- Show a short explanation when Broom intentionally blocked removal.
- Add an optional expandable list of remaining paths and reasons.
- Ensure notifications and summary screens do not imply total success when blocked items remain.

#### Primary files

## Exit Criteria

- Teams regression is covered and `UBF8T346G9.com.microsoft.teams` is suppressed when `com.microsoft.teams2` is installed and owns the container.
- Apple-managed `group.com.apple.*` containers never appear in generic leftover results.
- Embedded helper bundle IDs like `com.microsoft.autoupdate.fba` are suppressed when their parent app is installed.
- Directory candidates with recent descendant activity do not appear as leftovers even if the top-level folder timestamp is old.
- Apple-managed account files like `MobileMeAccounts.plist` never appear in generic leftover results.
- Container-only candidates do not present as medium confidence without stronger corroborating signals.
- Uninstall detail and uninstall confirmation are backed by the same editable plan.
- Cleaner and uninstaller completion states show blocked counts and reasons.
- CI produces code coverage output.
- A UI smoke suite runs in CI for the main navigation and primary entry points.
